# Registry, Forum, and Embedded UI — End-to-End Guide

This document explains how the three Phase-5 subsystems work together and how to
use them as an operator, contributor, or end user.

## Overview

```
Git repository (source of truth)
    |
    v
registry/index.json     <-- generated by CI, never hand-edited
    |
    +---> Forum (SQLite)    <-- populated by Reindex (FORM-05); ephemeral social layer
    |
    +---> CLI embed UI      <-- SvelteKit build, served locally by `debateos compose --serve`
    |
    +---> GitHub Pages      <-- separate BASE_PATH=/debateos build, zero-backend browse
```

The registry index is the **single source of truth** for all point metadata.
The Forum only stores what it reads from the index plus ephemeral social signals
(ratings, subscriptions, conflict threads). The embedded UI can be served entirely
offline — the Forum is additive, never required.

## Core Vocabulary

| Term            | Meaning                                                                            |
|-----------------|------------------------------------------------------------------------------------|
| **Opinion**     | An atomic, named configuration decision (e.g. "use Hyprland compositor")          |
| **Point**       | A curated bundle of opinions maintained by a Curator (the debate entry)            |
| **Speech**      | A user's personal composition of points from multiple curators                     |
| **Debate**      | The CLI visual conflict-resolution process — layering points and resolving clashes |
| **Foundation**  | The base OS (Arch, Debian, …) — interchangeable infrastructure, not identity      |
| **Curator**     | Maintainer of a Point; named identity, accountable for their choices               |
| **Translator**  | Foundation-specific code that turns an abstract Speech into an install artifact    |

## Registry Index

The registry index (`registry/index.json`) is generated automatically by CI on
every merge to `main` that touches a `registry/` subdirectory.

### Structure

```json
{
  "schema": "debateos-registry-index/v1",
  "generated_at": "2026-01-15T12:00:00Z",
  "points": [
    {
      "id": "hyprland-wayland",
      "name": "Hyprland Wayland Compositor",
      "intent": "Modern tiling Wayland compositor with animations",
      "curator": "mikkel0s",
      "members": ["hyprland-config", "waybar", "rofi-wayland"],
      "foundation_compat": [
        {"foundation": "arch", "compatible": true},
        {"foundation": "debian", "compatible": false, "missing": ["hyprland-aur"]}
      ],
      "commit_date": "2026-01-10T00:00:00Z",
      "tags": ["compositor", "wayland", "tiling"]
    }
  ]
}
```

### Adding a New Point

1. Create `registry/<point-id>/point.yaml` with the required fields.
2. Open a Pull Request — CI validates the schema and regenerates the index.
3. The Forum picks up the new point on next `forumctl reindex`.

## Forum (Optional Social Layer)

The Forum is a lightweight Go HTTP service backed by SQLite with FTS5 full-text
search. It provides:

- Browse and search all points by keyword or foundation
- Subscribe to points to track curator updates
- Rate points (1–5 stars) after GitHub OAuth sign-in
- Conflict threads for operators resolving incompatible points

### Forum is optional — Invariant 4

The core pipeline (`debateos compose` → resolve → build) **never contacts the Forum**.
The Forum is strictly additive. This is verified by `scripts/forum-offline-check.sh`
which asserts the pipeline produces correct output with no forum process running.

To verify locally:

```bash
bash scripts/forum-offline-check.sh
# Expected: INVARIANT-4 PASS
```

### Starting the Forum

```bash
go run ./forum/cmd/forumctl serve
# or, after building:
forumctl serve
```

Defaults: listens on `:8080`, database at `forum.db` (current directory).

### Recovering from Total DB Loss (FORM-05)

If the database is deleted or corrupted, recover all point metadata in seconds:

```bash
forumctl reindex --registry registry/index.json
```

Ephemeral social state (ratings, subscriptions) is not recoverable from the index —
only point metadata is stored there. This is by design: the index is git-derived
and does not contain user data.

### GitHub OAuth Setup (FORM-03)

The Forum requires GitHub OAuth for any write operation (ratings, subscriptions,
conflict threads). Read access (browse, search) is always public.

1. Register an OAuth app at <https://github.com/settings/applications/new>
   - Application name: `DebateOS Forum`
   - Homepage URL: your forum URL
   - Authorization callback URL: `https://<your-host>/oauth/callback`
2. Set environment variables:
   ```bash
   export GITHUB_CLIENT_ID=<your-client-id>
   export GITHUB_CLIENT_SECRET=<your-client-secret>
   export GITHUB_REDIRECT_URL=https://<your-host>/oauth/callback
   ```
3. Restart the Forum service.

Security properties (enforced in code):
- GitHub OAuth only — no native accounts, no passwords stored
- Access token used once for user ID lookup, then discarded (T-05-14)
- Session cookie is `httpOnly` + `SameSite=Lax` (T-05-12)
- OAuth state parameter validated with `crypto/rand` nonce (T-05-13 CSRF prevention)

## Embedded UI

The DebateOS UI is a SvelteKit single-page application built with a dual-delivery
strategy (T-05-18 mitigation):

| Build Target       | BASE_PATH           | Output            | Use                        |
|--------------------|---------------------|-------------------|----------------------------|
| GitHub Pages       | `/debateos`         | `dist/pages/`     | Static hosting on gh-pages |
| CLI embed          | `` (empty)          | `cli/embed/web/`  | Served locally by the CLI  |

### Serving Locally (Offline)

```bash
debateos compose --serve
# Serving DebateOS UI at http://localhost:8080
# Press Ctrl+C to stop.
```

The embedded build is committed under `cli/embed/web/` and includes the compiled
WASM resolver (`debateos.wasm`, ~4 MB). The UI runs entirely in-browser — no
server-side rendering, no Forum connectivity required.

Custom port:

```bash
debateos compose --serve --addr :3000
```

### Rebuilding the UI Locally

If you modify the SvelteKit source under `web/`:

```bash
bash scripts/build-ui-dual.sh
```

This script:
1. Compiles the Go WASM binary (`GOOS=js GOARCH=wasm`)
2. Runs `npm run build` twice — once per delivery target
3. Copies the embed build into `cli/embed/web/`
4. Copies the Pages build into `dist/pages/`

Skip the GitHub Pages build (e.g. on CI or if only testing the CLI embed):

```bash
SKIP_PAGES=1 bash scripts/build-ui-dual.sh
```

### GitHub Pages Deployment

The Pages build (`dist/pages/`) is deployed from the `gh-pages` branch via the
standard GitHub Actions Pages workflow. Live deployment is handled by CI — see
the `gh-pages` branch for the current live build.

## Operating in Harmony

The three subsystems are designed to complement rather than depend on each other.
A typical operator journey:

| Goal                             | What to run                                   |
|----------------------------------|-----------------------------------------------|
| Preview a local speech           | `debateos compose --dir /path/to/speech`      |
| Browse points offline            | `debateos compose --serve` → localhost:8080   |
| Browse points with social data   | Deploy Forum, visit `https://forum.example.com` |
| Recover Forum after data loss    | `forumctl reindex --registry registry/index.json` |
| Add a new point to the registry  | Add `registry/<id>/point.yaml`, open PR       |
| Regenerate registry/index.json   | CI triggers on PR merge; or run `bash scripts/generate-index.sh` |

The foundation is always Arch or Debian or whatever base you choose. The opinions
are what matter. The Debate never ends — it just gets more refined.
