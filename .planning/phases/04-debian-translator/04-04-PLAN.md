<objective>
Fix the ONE genuine infrastructure Arch-leak (D-DEB-03 Finding 5, VERIFIED in 04-RESEARCH.md):
cli/build/build.go hardcodes `translators/arch/translate` (line ~146), `arch-profile` (line ~141),
and the `--profile` default `vanilla-arch` (line ~81). This makes `debateos build` fail for any
non-Arch speech. The CLI layer must be foundation-agnostic — the user declares the foundation in
their speech.

Refactor (04-RESEARCH.md Pattern 5): introduce a data-driven `foundationRegistry` map keyed on
`rs.Foundation` (the ResolvedSpeech.Foundation string, VERIFIED present in resolver/resolve/explanation.go).
Each entry: {TranslateBin, ProfileDir, DefaultProfile}. arch → {translators/arch/translate, arch-profile,
vanilla-arch}; debian → {translators/debian/translate, debian-profile, debian}. The --profile flag default
becomes "" and resolves from the registry's DefaultProfile when empty (backward-compatible: explicit
--profile vanilla-arch still works; resolved open question item 3 + Pitfall 6).

Also author docs/arch-leak-audit.md — the COMPLETE DEB-03 audit (D-DEB-03): all 6 findings from
04-RESEARCH.md §Arch-Leak Audit with their type (genuine leak / intentional abstraction /
correctly-isolated) and action. Finding 5 (this build.go fix) is the only schema/code change; the
others are documented as correctly-isolated or intentional-abstraction (sig_level enum, install_phase
enum, mkinitcpio/limine tokens, keyring field interpretation).

Purpose: prove the abstraction is foundation-agnostic end-to-end and document the leak audit (DEB-03).
Output: foundation-aware build.go + go tests + docs/arch-leak-audit.md.
</objective>
