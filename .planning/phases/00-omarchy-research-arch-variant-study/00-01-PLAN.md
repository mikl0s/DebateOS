    Classify each opinion by what it DOES (its category from the 00-RESEARCH Category Taxonomy: package-install, package-removal, config-dotfile, service-enable, kernel-boot-param, theming, keybinding, hardware-conditional, arbitrary-script, user-group, sysctl-param, custom-repo, npm-global-install, display-manager, bootloader-config, snapshot-policy, firewall-rule, dns-config, font-install, mime-type), NOT by which install/ subdir it lives in (Pitfall 1 — many config scripts install packages or enable services).     - Base packages: do NOT record the 155-package list as one entry (Pitfall 2).     - Hardware-conditional: each of the ~34 hardware config scripts is a hardware-conditional opinion. Record the exact condition predicate (DMI / PCI ID / lspci pattern via the omarchy-hw-* helpers or inline grep) in a `condition:` field. Use `grep -rn "omarchy-hw-\|lspci\|dmi\|dmidecode\|/sys/class/dmi" /tmp/omarchy/install/config/hardware/`.
    - First-run: enumerate all install/first-run/*.sh (wifi, gnome-theme, firewall, dns, gdk-scale, etc.) and mark each `execution-phase: first-run` (Pitfall 3, RESEARCH open question 5).
    - Custom repo: record the `[omarchy]` repo (SigLevel = Optional TrustAll, pkgs.omarchy.org, keyring omarchy-keyring) and the hardware-conditional `arch-mact2` repo (SigLevel = Never) as custom-repo opinions with their trust level captured (Security Domain findings, threat T-00-RH).     - Themes: catalog all 21 theme directories under themes/ as theming opinions carrying file-asset payloads.
    - Migrations: sample 10-15 only; do NOT enumerate all 313 (Pitfall 5). Note the count + pattern for the Plan 04 open-questions deliverable; do not create OM entries for them here.
