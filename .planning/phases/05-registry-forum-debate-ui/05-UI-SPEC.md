# Phase 5 — UI Design Contract

> Visual and interaction contract for the DebateOS Debate UI (SvelteKit + Tailwind).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

No existing design system was detected (web/ is greenfield). shadcn is not applicable —
the stack is SvelteKit + Tailwind, not Next.js/React. All design tokens are defined
here as a Tailwind theme extension. All assumptions made in lieu of interactive answers
are tagged [ASSUMED].

## Spacing Scale

Declared values — 8-point scale, all multiples of 4:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, status dot margins, inline tight padding |
| sm | 8px | Between list items, button inner vertical padding |
| md | 16px | Default element spacing, card inner padding |
| lg | 24px | Section padding, pane header height offset |
| xl | 32px | Layout column gaps, modal inner padding |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top/bottom padding |

Exceptions:
- Touch targets for keyboard-navigable pane controls: minimum 44px height (WCAG 2.5.5 AAA target; satisfies 2.5.8 AA minimum 24px with margin). Applied to Pane header grab area and conflict action buttons.
- Foundation base strip: 56px fixed height (foundation identity bar; does not need to be a touch target multiple).

### 60/30/10 allocation

- **60% dominant** — `surface-base`: body background, the "stage" behind all panes.
- **30% secondary** — `surface-card`: sidebar panels, point cards, modal backgrounds, nav.
- **10% accent** — `accent-brand`: reserved exclusively for:
  - Primary CTA button fill ("Start Debating", "Apply Resolution", "Proceed to Build")
  - Active pane border ring (the selected/dragged pane indicator)
  - Focus rings (keyboard navigation)
  - Subscribe button in Forum discovery

`accent-brand` is NOT used for: tags, metadata text, conflict indicators, general links, or decorative elements.

### Conflict color rules (non-color-redundant encoding — mandatory)

Color alone is insufficient for accessibility. Every conflict state carries a second cue:

| State | Color | Icon | Label / ARIA |
|-------|-------|------|--------------|
| Hard conflict | `conflict-hard` (#ef4444) | `AlertTriangle` (lucide) | "Hard conflict" text label visible in overlap region |
| Compatible overlap | `conflict-compat` (#22c55e) | `CheckCircle2` (lucide) | "Compatible" text label visible in overlap region |
| Nice-to-have drop warning | `conflict-warn` (#f59e0b) | `Info` (lucide) | "Will be dropped" text label |
| Hardware mismatch | `conflict-warn` (#f59e0b) | `Cpu` (lucide) | "Hardware mismatch" text label |
| Patch available | `accent-brand` (#6366f1) | `Puzzle` (lucide) | "Patch available" text label |
| No conflict | — | — | No indicator rendered |

ARIA: each conflict overlay region has `role="status"` and `aria-label="[State]: [Opinion A] vs [Opinion B]"`.

Contrast check targets:
- `text-primary` on `surface-base` dark: #f1f3f9 / #0f1117 = 17.5:1 (exceeds AA + AAA)
- `conflict-hard` icon on `surface-pane` overlay: rendered at opacity 1 over dark base — verify ≥3:1 for non-text (WCAG 1.4.11)
- `accent-brand` on `surface-base` dark: #6366f1 / #0f1117 — minimum 3:1 for interactive elements; if < 4.5:1 add text label alongside (handled by always labelling CTA buttons)

## Compose / Debate Interaction Model

### States

| State | Trigger | UI Representation |
|-------|---------|-------------------|
| **Empty** | No panes added | FoundationBar shown; PaneStack shows EmptyPaneState component; "Add Your First Point" CTA visible |
| **Composing** | ≥1 pane added, no conflicts | Panes render with glass overlay; LiveResolveIndicator shows "Resolved" idle badge; no ConflictOverlay rendered |
| **Conflict** | WASM resolve() returns Dropped[] or hard-conflict Explanation | ConflictOverlay rendered on affected opinion rows; ResolutionPanel shows conflict details; BuildReadyBanner hidden |
| **Resolving** | User clicks Drop / Replace / Apply Patch action | WASM called again with updated input; LiveResolveIndicator spins; ConflictOverlay updates |
| **Resolved** | All hard conflicts cleared (Applied[] only, no Dropped[]... or only deliberate drops) | BuildReadyBanner shown in ResolutionPanel |
| **Build-Ready** | User confirms "Proceed to Build" | Speech export screen rendered with resolved YAML + CLI command |

### Interaction sequence (canonical happy path)

1. User arrives at `/` — sees FoundationStage with foundation selector (arch / debian) and empty pane stack.
2. User selects foundation → FoundationBar populates.
3. User searches/browses points → adds one or more to PaneStack via "Add to Debate" button.
4. On each add, UI calls `debateosResolve(JSON.stringify({ speech, opinions, hardware }))` via WASM.
5. If `result.Explanations` contains hard conflicts → ConflictOverlay renders on affected opinion rows in PaneStack.
6. ResolutionPanel shows first unresolved conflict's Explanation.Text + available actions.
7. User selects Drop / Replace / Apply Patch → UI updates speech input → calls WASM again → overlay updates.
8. When no hard conflicts remain → BuildReadyBanner appears.
9. User clicks "Proceed to Build" → navigates to `/export` with resolved speech YAML + `debateos build` command.

### WASM call contract

```typescript
// Input: JSON string matching resolver/wasm resolveInput shape
const input = JSON.stringify({
  speech: Speech,         // resolver.Speech
  opinions: Opinion[],    // resolver.Opinion[] — all opinions referenced by speech
  hardware: HardwareProfile  // resolver.HardwareProfile
});

// Call: synchronous return (WASM is single-threaded)
const raw = window.debateosResolve(input);

// Output: JSON string matching resolveOutput shape
// { result?: string, error?: string }
// result is canonical JSON of ResolvedSpeech when present (even on hard conflict)
const out = JSON.parse(raw);
const resolved: ResolvedSpeech = JSON.parse(out.result);
// out.error is set alongside result on hard conflicts (non-nil error = at least one hard conflict)
```

WASM call is made on every pane add/remove and every conflict resolution action.
Debounce: 150ms after last user action before calling resolve to avoid hammering WASM
during rapid drag-reorder. LiveResolveIndicator shows "Resolving..." during debounce + call.

### Pane drag-and-reorder

Panes are reorderable via drag-and-drop (mouse + keyboard). Keyboard: Tab to pane header,
Space to "grab", arrow keys to move, Space/Enter to drop. Uses native HTML5 drag API
with Svelte actions; no third-party DnD library [ASSUMED: keeps bundle small for WASM
co-bundle].

## Conflict Visualization Spec

### Overlay rendering model

ConflictOverlay is rendered as an absolutely-positioned element spanning the opinion rows
involved in the conflict, with `mix-blend-mode: normal` at `opacity: 0.18` for the
fill, and a solid 2px border at `opacity: 1` plus the ConflictBadge icon+text at full
opacity.

This gives:
- Subtle color wash across the affected rows (colour cue)
- Solid border delineating the conflict zone
- Icon + text badge in the center of the overlay (redundant encoding)

### States rendered

```
Hard conflict (rule2 / Explanation.Rule == "rule2"):
  Background: conflict-hard at 0.18 opacity
  Border: 2px conflict-hard solid
  Icon: AlertTriangle (filled, conflict-hard color)
  Label: "Hard conflict" in 14px/600
  Actions: [Drop A] [Drop B] [Apply Patch] (if PatchOffered is non-empty)

Compatible overlap (Explanation.Rule == "no-conflict", multiple panes share opinion):
  Background: conflict-compat at 0.12 opacity
  Border: 2px conflict-compat solid dashed
  Icon: CheckCircle2 (conflict-compat)
  Label: "Compatible" in 14px/600

Nice-to-have drop (rule1 / Explanation.Rule == "rule1"):
  Background: conflict-warn at 0.14 opacity
  Border: 2px conflict-warn solid
  Icon: Info (conflict-warn)
  Label: "Will be dropped" in 14px/600
  Subtext: Explanation.Text (body 16px/400, text-secondary)

Hardware mismatch (Explanation.Rule == "hardware-skip"):
  Background: conflict-warn at 0.14 opacity
  Border: 2px conflict-warn dashed
  Icon: Cpu (conflict-warn)
  Label: "Hardware mismatch"
  Subtext: Explanation.AlternativeSuggestion (if non-empty)

Patch available (Explanation.PatchOffered non-empty):
  Additional badge: Puzzle icon + "Patch available" in accent-brand
  Rendered alongside the conflict overlay, not replacing it
```

### Explanation rendering (ResolutionPanel)

Each `Explanation` from `ResolvedSpeech.Explanations[]` is rendered as an ExplanationCard:

```
ExplanationCard
├── Rule badge          — pill: "Rule 1" / "Rule 2" / "Rule 3" / "Rule 4" / "Hardware" / "Ordering"
├── Explanation.Text    — body 16px/400, full text verbatim (resolver owns the prose)
├── Opinions involved   — OpinionsInvolved[] rendered as opinion-name chips
├── Kept row            — green check icon + opinion names (if Kept[] non-empty)
├── Dropped row         — red X icon + opinion names (if Dropped[] non-empty)
├── Patch row           — Puzzle icon + PatchOffered ID + "Apply" button (if PatchOffered non-empty)
└── Trust warning       — amber warning banner (if TrustWarning non-empty; verbatim text)
```

## Forum-Offline Degradation

Forum availability is optional (invariant 4). When Forum is unreachable:

### Detection

On page load, the UI issues a `HEAD` request to the Forum health endpoint with a 3-second
timeout. If the response is not 2xx within timeout, Forum is marked offline for that session.
Re-check every 5 minutes passively [ASSUMED: simple interval; no websocket needed for v1].

### ForumOfflineBanner

Rendered at top of Point Browser and Point Detail screens (NOT on the Debate/Compose screen —
the compose path never requires Forum):

```
Background: surface-forum-offline (#2d1f0e dark / #fef3c7 light)
Icon: WifiOff (lucide, conflict-warn amber)
Text: "The Forum is offline. Showing registry index only — search and ratings are
       unavailable. Your debate and build are unaffected."
Dismiss: X button (persists dismissal in sessionStorage)
```

### Feature degradation table

| Feature | Forum Online | Forum Offline |
|---------|-------------|---------------|
| Point/speech browse | FTS5 search + sort | Client-side filter over static registry JSON |
| Tag filter | Dynamic from Forum | Static tag list from registry index |
| Popularity sort | Live counts | Unavailable — sort hidden, freshness used |
| Ratings display | Shown | Hidden (no error — simply absent) |
| Rate a point | Enabled | CTA replaced with "Ratings unavailable offline" |
| Conflict threads | Shown | Replaced with GitHub link |
| Follow curator | Enabled | Button disabled, tooltip "Requires Forum" |
| Compose a debate | Full function | Full function — zero Forum dependency |
| Resolve conflicts | Full function | Full function — WASM only |
| Export / build | Full function | Full function |

## Accessibility Baseline

### Keyboard navigation

- All interactive elements reachable by Tab in logical DOM order.
- PaneStack supports keyboard drag-reorder (Space to grab, Arrow to move, Space/Enter to drop).
- ConflictActions (Drop / Replace / Apply Patch) buttons are tab-stops within the conflict overlay.
- No keyboard trap anywhere. ESC always dismisses overlays and inline confirmations.
- Focus ring: 2px solid `accent-brand` at 2px offset — visible on both dark and light backgrounds.

### Screen reader

- Each PaneCard: `role="region"` with `aria-label="[Point name] pane"`.
- Each ConflictOverlay: `role="status"` with `aria-label="[State]: [Opinion A name] conflicts with [Opinion B name]"`.
- ExplanationCard rendered in a `<section>` with a heading (heading 20px/600) so it is navigable by heading.
- LiveResolveIndicator: `role="status"` aria-live="polite" — announces "Resolving…" and "Resolved" without interrupting.
- ForumOfflineBanner: `role="alert"` aria-live="assertive" on first appearance only; dismissible.
- WasmLoadGate: `role="status"` aria-live="polite", "Loading the resolver…".

### Color blindness (redundant encoding — mandatory)

Red/green conflict encoding must never be the sole cue (deuteranopia/protanopia affects ~8% of males).
This is met by the triple encoding defined in §Conflict Visualization Spec:
- Color fill + border (colour cue)
- Icon (shape cue: AlertTriangle vs CheckCircle2 — distinctly different shapes, not just color)
- Text label (text cue: "Hard conflict" vs "Compatible")

Amber (`conflict-warn`) is used for warnings to avoid false positives for red-green deficiency users.

### Contrast targets

| Pairing | Target | Standard |
|---------|--------|----------|
| Body text on surface-base (dark) | ≥ 7:1 | WCAG AA (4.5:1) and AAA (7:1) |
| Label text on surface-card (dark) | ≥ 4.5:1 | WCAG AA |
| Accent-brand button text on accent-brand bg | ≥ 4.5:1 | WCAG AA — verify: white text on #6366f1 |
| Conflict-hard icon on surface-pane | ≥ 3:1 | WCAG 1.4.11 (non-text contrast) |
| Conflict-compat icon on surface-pane | ≥ 3:1 | WCAG 1.4.11 |
| Focus ring (accent-brand) vs adjacent bg | ≥ 3:1 | WCAG 1.4.11 |

### Responsive layout

- Minimum viewport: 768px (tablet landscape). The Debate UI is a power-user tool; sub-768px degrades to a notice: "The Debate works best on a wider screen. You can still browse points on mobile."
- Desktop (≥1280px): three-column layout — left nav/browse sidebar | center PaneStack | right ResolutionPanel.
- Tablet (768–1279px): two-column — PaneStack | ResolutionPanel; Browse is a slide-over panel.
- Motion: respects `prefers-reduced-motion`. Pane animations and ConflictOverlay fade are `transition-none` when reduced motion is preferred.

## Verifiable Design Assertions

The following assertions are checkable by gsd-ui-checker and gsd-ui-auditor against
the implemented `web/` output. Each assertion is testable via DOM inspection,
computed styles, or Playwright snapshot.

### A1 — Conflict triple encoding

**Assert:** Every ConflictOverlay element with `data-conflict-state="hard"` has:
1. A computed background-color matching #ef4444 (possibly at reduced opacity via rgba)
2. A child element with `data-icon="AlertTriangle"` (or aria-label containing "AlertTriangle")
3. A child element with visible text matching /hard conflict/i

**Failure mode:** Relying on color alone for conflict communication.

### A2 — Touch target minimum

**Assert:** All `.pane-header` elements and `.conflict-action-btn` elements have
`offsetHeight >= 44`.

**Failure mode:** Conflict action buttons too small for touch interaction.

### A3 — WASM-only compose path

**Assert:** Loading the Debate screen (`/debate`) with no Forum reachable produces
a working compose experience — WASM resolver initializes, points can be added,
resolve() returns a ResolvedSpeech. No network call to Forum domain is required.

**Playwright test:** Block Forum origin, load `/debate`, add example point, assert
`ResolvedSpeech.Applied.length > 0`.

### A4 — Forum-offline banner

**Assert:** When Forum health check fails, ForumOfflineBanner appears on `/browse`
with `role="alert"` and text matching /forum is offline/i. Banner does NOT appear
on `/debate`.

### A5 — Accent reserved

**Assert:** `accent-brand` color (#6366f1 dark) appears ONLY on:
- Primary CTA buttons (class contains `btn-primary` or similar)
- Active pane border ring (`data-pane-active="true"`)
- Focus rings (`:focus-visible` outline)
- Subscribe/Follow button

No other elements compute this color as their primary background or text color.

### A6 — Terminology compliance

**Assert:** No user-facing string (excluding code blocks and YAML display) contains
the words "config", "preset", "package set", or "distro". All UI copy uses the
canonical DebateOS vocabulary: "Opinion", "Point", "Speech", "Debate", "Foundation",
"Translator".

**Test:** Playwright `page.getByText()` queries for forbidden terms in visible text nodes.

### A7 — Typography scale

**Assert:** No font-size value outside {13, 14, 16, 20, 28}px appears in the
application (excluding browser default user-agent styles on native elements).
No font-weight outside {400, 600} appears (with 500 as the sole tolerance for
system-rendered intermediate weights).

### A8 — Spacing multiples

**Assert:** All explicit padding and margin values are multiples of 4px.
The one exception allowed: border-radius values.

### A9 — Explanation text verbatim

**Assert:** When the WASM resolver returns an Explanation with a non-empty Text field,
that exact string is rendered verbatim in the ExplanationCard. No UI prose is
substituted for the resolver's explanation text.

**Test:** Load known fixture speech, call debateosResolve, compare Explanation[0].Text
to the rendered ExplanationCard text content.

### A10 — Reduced motion

**Assert:** With `prefers-reduced-motion: reduce` set in the test browser, no CSS
`transition` or `animation` property with a duration > 0ms applies to pane reorder
or ConflictOverlay fade elements.
