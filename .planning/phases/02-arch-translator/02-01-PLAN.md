<objective>
Establish the Arch translator Python package, its test harness (pytest), and the two foundational TDD modules: the input-contract loader + capability gate (ARCH-03) and the build-manifest builder (ARCH-01 foundation). This plan owns the translator's data layer — turning a resolved speech (decisions) plus opinion bodies (payloads) into a single typed BuildManifest, and failing loudly when a required opinion needs an undeclared capability.

Purpose: The downstream profile emitter (Plan 02) and all slow gates (Plan 05) consume BuildManifest. Getting the contract and capability gate right first means the rest of the phase builds against a stable, tested data structure rather than re-deriving it.

Output: `translators/arch/` package skeleton + `capabilities.json` + `capabilities.py` + `contract.py` + `manifest.py`, all GREEN under pytest, with RED-first commits.
</objective>

<artifacts_this_phase_produces>
This plan establishes the translator INPUT CONTRACT, which Phase 3's CLI depends on:

- **ResolvedSpeech JSON** (`resolver/resolve.ResolvedSpeech` via `CanonicalJSON`) carries ONLY decision IDs: `applied`, `skipped`, `dropped`, `install_order` (all `[]OpinionID`), plus `explanations`. It does NOT carry opinion bodies.
- Therefore the translator contract is TWO inputs: (1) the resolved.json (authoritative decisions + order), (2) an opinion-body source providing each applied opinion's full `Opinion` struct (packages, file_assets, services, …) keyed by ID. The opinion-body source for examples/omarchy is the authored opinion YAML/JSON; the loader accepts a directory of opinion files OR a single JSON array.
- `capabilities.json` schema: `{ "capabilities": ["token", ...] }` — the declared supported set.
</artifacts_this_phase_produces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| resolved.json + opinion bodies → generator | Locally-authored data in Phase 2; attacker-controlled in Phase 5 registry context. Treat opinion payload strings as data, never as code. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | Opinion payload strings flowing into manifest | mitigate | Serialize all payloads as JSON (to_dict / build-manifest.json); installer reads JSON, never shell-interpolates raw strings (research Pitfall 6) |
| T-02-02 | Spoofing | custom_repos with sig_level=Never | mitigate | Manifest captures trust_warnings; surfaced as pacman.conf comment in Plan 02; resolver already emits TrustWarning (T-01-10) |
| T-02-03 | Elevation | Required opinion needing undeclared capability silently passing | mitigate | check_capabilities raises CapabilityError at composition time (ARCH-03 / SC-3) BEFORE any profile or build step |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan; pytest+PyYAML are Arch-official (research Package Legitimacy Audit — all OK) |
